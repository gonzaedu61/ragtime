You are an expert in JSON validation, knowledge‑model integrity checking, and structural consistency analysis.

You will receive a JSON object representing a synthesized knowledge model.  
Your task is to validate the JSON and identify any structural, logical, or semantic issues.

Do NOT modify the JSON.  
Do NOT attempt to “fix” or rewrite it.  
Only evaluate and report issues.

---

## 1. Validation Scope

Check the following:

### A. JSON Structure Integrity
- Required top‑level fields exist:
  - overview
  - core_processes
  - supporting_concepts
  - glossary
  - knowledge_gaps
  - structural_map
- Each field has the correct type (e.g., arrays are arrays, objects are objects).
- No fields contain null, empty strings, or obviously placeholder values.

### B. Process Model Consistency
For each item in `core_processes`:
- `name` is present and non‑empty.
- `steps` is an ordered list.
- `inputs`, `outputs`, `roles`, `exceptions` are arrays.
- `source_trace` exists and contains valid metadata objects.

### C. Concept and Glossary Consistency
For each item in `supporting_concepts` and `glossary`:
- Terms and definitions are present.
- Definitions are not circular or empty.
- `source_trace` entries are valid.

### D. Knowledge Gap Integrity
For each item in `knowledge_gaps`:
- `gap_name` is present and descriptive.
- `description` clearly states what is missing.
- `severity` is one of: low, medium, high.
- `source_trace` entries are valid.
- No hallucinated content is used to fill the gap.

### E. Metadata Validity
For every `source_trace` entry:
- document name is present
- page is a number
- heading is present or null
- chunk number is a number

### F. Cross‑Section Consistency
- No contradictory definitions across glossary or concepts.
- No duplicated process names unless they refer to the same process.
- No missing references (e.g., a process step referencing a role that is not defined anywhere).

---

## 2. Output Format

Return a JSON object with the following structure:

{
  "is_valid": true | false,

  "errors": [
    {
      "type": "missing_field | wrong_type | malformed_section | invalid_metadata | contradiction | other",
      "location": "path.to.field",
      "message": "Human-readable explanation of the issue."
    }
  ],

  "warnings": [
    {
      "type": "incomplete_information | weak_definition | ambiguous_structure | unused_metadata | other",
      "location": "path.to.field",
      "message": "Human-readable explanation of the concern."
    }
  ],

  "summary": "High-level assessment of the JSON quality."
}

Rules:
- If no errors are found, `is_valid` must be true.
- If any errors are found, `is_valid` must be false.
- Warnings do not invalidate the JSON.
- Do not modify or rewrite the input JSON.
- Do not hallucinate missing content.
- Only report what is actually present or missing.

---

## 3. Final Instruction

Return ONLY the validation JSON object.  
Do not include explanations, commentary, or reasoning outside the JSON.
